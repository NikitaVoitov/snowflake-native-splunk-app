## What went well

1. **Epic 2 delivered a real integration path, not just UI wiring.**  
   Story 2.1 established the correct Native App pattern for outbound connectivity: runtime endpoint normalization, network rule updates, application specification updates, and a server-side gRPC readiness probe. That gave the project a working OTLP setup path grounded in Snowflake behavior.

2. **Security boundaries improved as the epic progressed.**  
   The epic consistently moved sensitive handling in the right direction: connection testing and certificate parsing run server-side, raw PEM material is kept out of `_internal.config`, and Story 2.3 completed the secret-backed persistence path rather than falling back to unsafe table storage.

3. **The onboarding experience became durable and navigable.**  
   Story 2.3 turned Getting Started into a live onboarding hub with task cards, sidebar progress, drill-down navigation, and state loaded from Snowflake-backed config rather than client-only state. That gives later epics a usable setup journey instead of placeholder scaffolding.

4. **Epic 1 preparation work paid off directly.**  
   The real collector endpoint, TLS materials, probe tooling, and SSH access prepared in the Epic 1 retrospective were actually used in Epic 2. The team did not have to guess about OTLP/TLS behavior once implementation started.

5. **Review cycles improved the quality of the final solution.**  
   Story 2.1 tightened stale-state reset behavior before closeout. Story 2.2 reconciled Save gating with product intent and improved timeout messaging. Story 2.3 completed the durable config and onboarding experience instead of leaving the epic half-wired.

## Friction and lessons

1. **External access approval is a product workflow concern, not just a backend detail.**  
   The application specification approval flow affects what the user sees and when a test can succeed. That means approval messaging, retry behavior, and host-switch handling must stay first-class UX concerns in future export-related work.

2. **Save gating and stale-state invalidation need explicit design, not incidental state management.**  
   This surfaced repeatedly across the epic. Endpoint edits, certificate edits, and saved-state hydration all affect whether the current UI state is trustworthy. The team benefited when those rules were spelled out and tested instead of inferred.

3. **Snowflake runtime diagnostics require careful interpretation.**  
   Epic 2 exposed that Python-level DNS results inside Snowflake can be misleading for real outbound connectivity. Diagnostic copy must stay tied to observed transport behavior and avoid overclaiming root causes from advisory signals.

4. **Forward-compatible onboarding shortcuts are useful, but they create replacement work.**  
   Epic 2 intentionally introduced interim Getting Started completion signals for later tasks so the onboarding flow could behave like a real product. That was the right decision, but it creates a clear handoff obligation for Epic 3 and later epics to replace those stubs cleanly.

5. **Platform-specific implementation patterns belong in shared project memory once discovered.**  
   The dynamic EAI provisioning pattern, app-spec accumulation behavior, secret-backed PEM handling, and Native App session/reset quirks are now proven knowledge. Future epics should treat these as baseline patterns, not rediscover them story by story.

## Breakthroughs worth keeping

1. **Dynamic EAI provisioning with a stable integration name.**  
   Keeping the EAI reference stable while updating the underlying network rule and application specification is the right Native App pattern for this project.

2. **Application-spec accumulation reduced repeat approval pain.**  
   Preserving previously approved `HOST_PORTS` while keeping the network rule focused on the active endpoint is a practical compromise that improves usability without weakening the active runtime boundary.

3. **Secret-backed PEM persistence clarified the security model.**  
   By the end of the epic, the project had a clean split: non-secret destination settings in `_internal.config`, secret material in app-owned secrets, and UI hydration that respects that boundary.

4. **Getting Started is now a real product surface.**  
   The onboarding hub can carry meaningful completion state and page-to-page drill-downs. That makes it a useful contract for later epics instead of a throwaway placeholder.

---

## Technical debt and carry-forward items

1. **Getting Started still contains interim completion signals for later epics.**  
   Task 2, Task 3, and Task 4 currently rely on temporary config-backed markers introduced so Epic 2 could ship a workable onboarding flow. Epic 3 and later work must replace those signals with real completion logic without breaking progress tracking.

2. **Sidebar badge styling is implementation-dependent.**  
   The Getting Started badge uses DOM-targeted CSS against the current Streamlit sidebar structure. That is acceptable for now, but it should be re-verified whenever Streamlit behavior changes.

3. **Provider-side reset behavior can leave open sessions stale.**  
   The dev reset flow clears durable state correctly, but active sessions may need rerun or restart to reflect the reset immediately. This is manageable, but the team should remember it during validation.

4. **Artifact hygiene still needs attention.**  
   Epic 2 implementation artifacts captured important discoveries, but story record quality is not perfectly consistent. Future epics should keep the same technical depth while tightening template cleanup and closeout completeness.

## Epic 3 readiness

Epic 3 remains the correct next step and does **not** require a fundamental replan.

The project now has the prerequisites Epic 3 needs:

- A working Splunk Settings flow that writes durable config.
- A real Getting Started hub that can reflect progress from durable state.
- An existing Telemetry Sources page and onboarding hooks ready to be replaced with real source-selection logic.
- Established config-key conventions for `pack_enabled.*` and `source.<name>.*`.

### What Epic 3 can build on safely

- `otlp.endpoint` persistence is already real, so source configuration can assume destination setup exists.
- Getting Started drill-down and return flows are already in place.
- The config/secrets split is already defined, so Epic 3 can focus on source state rather than reworking destination persistence.

### What needs attention before or during Epic 3

1. **Replace the temporary Task 2 completion signal with real pack/source completion logic.**  
   `pack_enabled.dummy` was a good bridge for Epic 2, but Epic 3 should retire it in favor of real source-selection state.

2. **Keep completion state DB-backed, not session-only.**  
   Epic 2 intentionally tied onboarding progress to Snowflake-backed config. Epic 3 should preserve that discipline for discovery, edits, and save flows.

3. **Preserve config-key conventions.**  
   Epic 3 should continue using the dotted key patterns already established in architecture and implementation (`pack_enabled.<pack_name>`, `source.<name>.view_fqn`, `source.<name>.poll_interval_seconds`, and related keys).

4. **Validate discovery and save flows against real account behavior.**  
   Epic 2 reinforced that Snowflake-specific behavior should be proven live. Source discovery, editable intervals, and persistence should follow the same real-environment validation standard.
