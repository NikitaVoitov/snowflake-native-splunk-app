## Friction and lessons

1. **Platform discoveries were repeated instead of captured in project rules.**  
   Key patterns like `st.data_editor` state management (never write edited DataFrame back), fragment architecture, widget key cleanup on navigation, and the RCR limitation were documented in story records but not systematically captured in cursor rules. This meant each new story (and each new AI agent session) had to re-learn patterns from story files rather than having them as baseline project knowledge. The fix is straightforward: after each story, update cursor rules with any significant platform or implementation patterns discovered during development.

2. **AI agents needed repeated guidance on tooling and workflow.**  
   The same instructions about which tools to use (Snow CLI, Playwright MCP, Cortex CLI), which venv to use for tests, how to deploy, and how to validate against the live account had to be repeated in story creation and development sessions. Discovered patterns should flow into cursor rules so AI agents inherit project conventions automatically, not rediscover them from scratch with every story.

3. **The Streamlit widget key cleanup behavior was a subtle, high-impact bug.**  
   Streamlit silently removes widget keys from `st.session_state` when a page is not rendered (navigating away). This caused pack toggles to reset to OFF after save→navigate→return, despite correct values in `_internal.config`. The root cause was non-obvious: the initialization code defaulted to `False` for missing keys, and the signature-based cache prevented a DB reload. The fix (restoring from a non-widget snapshot) was simple once understood, but the debugging cost was real. **Lesson:** Never use the same key for both widget state and canonical state in a multipage Streamlit app. Widget keys are transient.

4. **The `st.data_editor` has sharp behavioral edges that must be learned once and encoded.**  
   Across Stories 3.1 and 3.2, the team discovered multiple `st.data_editor` patterns that are not obvious from Streamlit docs: the double-toggle bug when writing edited DataFrame back, the `pinned=True` requirement for fixed-width columns, the editor internal state structure (`edited_rows`/`added_rows`/`deleted_rows`), and the CSS sibling-marker pattern for conditional row styling. These are now documented in the Streamlit design rules but were initially discovered through trial and error.

5. **Discovery latency for Event Tables (up to 45 min via ACCOUNT_USAGE) is a UX trade-off.**  
   The MVP uses `SNOWFLAKE.ACCOUNT_USAGE.TABLES` for event table discovery because it's the only viable account-wide approach from a Native App. The latency is acceptable for a configuration page but means newly created event tables won't appear immediately. The RCR investigation confirmed there's no faster alternative in warehouse runtime.

## Breakthroughs worth keeping

1. **`@st.fragment` architecture pattern for Native App pages.**  
   Place static chrome (CSS injection, `st.header`, `st.caption`, `st.info`) outside the fragment. Place all interactive widgets, data editors, toggles, and the footer inside a single `@st.fragment` function. Widget clicks rerun only the fragment body, not the entire page. This pattern eliminates the full-page greyout and avoids re-executing expensive queries on every click.

2. **Editor-overlay pattern for `st.data_editor` state.**  
   Never write the editor's return value back into the base DataFrame. Instead, read the editor's internal state from `st.session_state[editor_key]["edited_rows"]` and overlay edits onto the base DataFrame using `_effective_values(df, editor_key, column)`. This prevents the double-toggle bug and works for checkboxes, numbers, and any editable column.

3. **Widget key cleanup resilience via non-widget snapshots.**  
   Store canonical state in a non-widget session state key (e.g., `_SAVED_STATE_KEY`) and restore widget keys from that snapshot when they go missing after page navigation. This separates transient widget state from durable session state.

4. **Discovery signature-based caching with forced reload after save.**  
   Cache the discovery→load cycle using a signature of discovered sources. After save, clear the signature and set a reload flag to force a fresh DB read on the next rerun. This avoids redundant DB queries on widget clicks while ensuring save consistency.

5. **Config batch save for atomic multi-key writes.**  
   `save_config_batch()` writes multiple `_internal.config` keys in a single SQL round-trip, avoiding partial-save states when pack toggles and per-source settings must be persisted together.

6. **Event Table overlap exclusion across all persistence phases.**  
   The three-phase guard (load, capture, save) ensures Event Table sources never leak `overlap_minutes` into `_internal.config`, even if stale rows exist from earlier saves. This pattern — enforcing schema rules at every boundary, not just one — prevents subtle data contamination.

## Technical debt and carry-forward items

1. **Cursor rules do not yet capture Epic 3 platform discoveries.**  
   The RCR limitation, widget key cleanup behavior, `st.data_editor` patterns, and fragment architecture are documented in story records but not in cursor rules. This creates re-learning overhead for every new AI agent session. An explicit rule-update step after each story would prevent this.

2. **Event Table discovery latency (up to 45 min) is a known UX limitation.**  
   The MVP uses `SNOWFLAKE.ACCOUNT_USAGE.TABLES` which has inherent latency. The manual "Discover sources" button mitigates this by giving users control over when discovery runs. Real-time discovery via RCR is blocked on Snowflake container runtime GA for Native Apps.

3. **RCR procedure code is commented out in `setup.sql`.**  
   The RCR investigation left commented-out procedure code as a reference for future container runtime migration. This is intentional and should not be cleaned up until container runtime is available for Native App Streamlit.

4. **`st.set_page_config(layout="wide")` is set globally.**  
   Story 3.2 added wide layout with 3rem side padding for the Telemetry Sources page. This affects all pages since `set_page_config` is a global setting. Other pages may need CSS adjustments if the wide layout creates undesirable spacing.

5. **Batch size column is explicitly deferred.**  
   The architecture specifies a per-source batch size setting, but this was descoped from Epic 3. Epic 5 or a post-MVP story should add this when collection pipelines need configurable batch sizes.

## Key platform discoveries (Epic 3)

These discoveries should be encoded in cursor rules to prevent re-learning.

### 1. RCR (Restricted Caller's Rights) does not work from Streamlit warehouse runtime

**Finding:** Streamlit in Native Apps runs exclusively in warehouse runtime, which always executes as the app owner role. When the Streamlit session calls an RCR procedure, the "caller" identity is the app owner, not the consumer. `INHERITED CALLER` grants allow the procedure to use privileges the caller *already has* — they don't confer new ones. Since the app owner inherently lacks SELECT on consumer event tables, the RCR procedure sees nothing.

**Impact:** Any future feature that needs to access consumer objects (beyond what `IMPORTED PRIVILEGES` provides) cannot use RCR from Streamlit. Container runtime with `st.connection("snowflake-callers-rights")` would be required — currently in preview, not available for Native App Streamlit.

**Reference:** Story 3.2 Key Findings §1.

### 2. Streamlit removes widget keys from session state on page navigation

**Finding:** When a user navigates away from a Streamlit page, Streamlit removes all widget keys (e.g., `st.toggle` keys) from `st.session_state`. On return, these keys are missing and default to initial values. This is silent — no error, no warning.

**Fix pattern:** Store canonical state in a non-widget session state key. Restore widget keys from that snapshot when they go missing. Require the snapshot to exist before allowing cache-based early returns in load functions.

**Reference:** Story 3.2 Key Findings §3.

### 3. `TABLE_TYPE` for event tables is `'EVENT TABLE'` (two words)

**Finding:** `SNOWFLAKE.ACCOUNT_USAGE.TABLES` uses `TABLE_TYPE = 'EVENT TABLE'` (with a space), not `'EVENT'` or `'EVENT_TABLE'`. Verified via live SQL.

**Reference:** Story 3.2 Key Findings §2.
