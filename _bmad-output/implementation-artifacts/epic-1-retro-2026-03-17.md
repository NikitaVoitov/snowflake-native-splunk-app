## What went well

1. **The foundation is real, not theoretical.**  
   The install path, warehouse binding, Streamlit shell, and core state tables were validated against actual Snowflake behavior. That matters because this epic was full of platform-specific behavior that is easy to misunderstand from docs alone.

2. **We corrected the most important gap before closing the epic.**  
   Reopening Story 1.1 to finish warehouse binding was the right call. Epic 1 should only be considered done when the app installs, binds a warehouse, and the Streamlit UI can actually run queries. That is now true.

3. **We have a usable shell for the next epic.**  
   The Splunk settings page stub, Getting Started route, About dialog, and page routing give Epic 2 a solid place to build instead of forcing another round of shell work.

4. **We developed good instincts for validating Snowflake Native App behavior.**  
   The strongest progress came from checking the app in its real runtime context: verifying packaged Streamlit assets on stage, checking the installed app behavior in Snowsight, and validating internal objects from the correct execution context.

---

## Friction and lessons

1. **Snowflake Native App behavior still has sharp edges.**  
   Artifact mapping, installed-app visibility, and Streamlit runtime behavior all required live verification. The lesson is simple: for this project, important platform assumptions should be proven in the target environment early.

2. **Provider-side metadata is not always the source of truth.**  
   Some app-internal objects are not reliably visible from the provider-side perspective, even when they exist and work correctly. For internal objects, app-context verification is sometimes the only dependable check.

3. **Repeat deploy checks should be intentional, not automatic.**  
   Re-run and idempotency checks were useful during foundation work, especially for setup and packaging changes. They should remain part of the toolkit, but only when a change actually touches install, setup, deploy, or object-creation behavior. They should not become a blanket requirement for every UI iteration because that slows feedback without adding much value.

## Epic 2 readiness

Epic 2 is not just form wiring. It introduces real external validation and needs real test assets.

### Story 2.1 — OTLP endpoint and connection test

This story needs a **real OTLP-capable endpoint** that can accept connections from Snowflake. A mock text field is not enough. The work should be prepared around:

- A reachable OTLP collector or equivalent endpoint that supports **gRPC**.
- Real connection checks from the app to that endpoint, not just string validation.
- The Snowflake-side objects required for outbound connectivity, such as the appropriate network rules and related egress configuration.
- Target-side network access that allows inbound gRPC traffic from Snowflake to reach the collector successfully.

### Story 2.2 — Optional certificate and validation

This story needs a **real PEM certificate** for validation work. The checks should go beyond basic parsing and cover practical certificate validation concerns that matter for a real destination:

- PEM structure is valid.
- Certificate validity window is current.
- Hostname usage is compatible with the configured endpoint.
- The certificate is appropriate for the intended TLS use.

### Story 2.3 — Persist destination config and Getting Started hub

This story should deliver the **full Getting Started experience**, not a placeholder. That means:

- Task tiles for the four setup steps.
- Progress state and visual completion feedback.
- Drill-down behavior from tasks into the correct page or flow.
- Persistent completion state after Maya saves valid destination settings.

Because FR3 is delivered through Epic 2, Story 2.3 should leave the app with a **workable Getting Started page**, not just layout scaffolding. For tasks that depend on stories from later epics, Story 2.3 can use temporary UI placeholders for state progression so the page behaves like a real onboarding hub while the downstream features are still being built. A practical approach is:

- **Splunk settings** must be completed for real through the actual connection and save flow.
- The other tasks can use temporary state controls in Story 2.3, such as checkboxes or equivalent UI toggles, so their completion state updates the sidebar counter and the visual state of the Getting Started page.
- These temporary controls should be easy to replace later with real completion signals once the corresponding stories are implemented.

This note should be carried forward when Story 2.3 is created so the implementation intentionally delivers a usable onboarding experience in Epic 2 while remaining compatible with later epics.

Epic 2 therefore has external preparation work in addition to UI implementation. The foundation from Epic 1 is ready, but Epic 2 still needs real integration targets and real validation assets before it can be considered fully testable.

An additional advantage for Epic 2 is that development and troubleshooting can use direct access to the target collector environment. SSH access to the OTLP/gRPC-capable endpoint will make it possible to inspect collector logs and use OS-level CLI tools while validating real connection behavior during development.
